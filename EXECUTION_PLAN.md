# Qorax — Execution Plan

Цей документ — не нова архітектура, а **послідовність дій**, що
зводить воєдино вже написані документи в один порядок роботи.
Кожен пункт нижче посилається на розділ, де рішення вже прийнято —
тут не повторюється деталізація, тільки що робити і в якому порядку.

```
PRODUCT_VISION.md   → навіщо
PLATFORM.md         → як росте платформа
DATA_MODEL.md       → структура даних
PRICING.md          → монетизація
SECURITY.md         → авторизація/RLS
API_CONTRACTS.md    → конвенція ендпоінтів
MODULE_ROADMAP.md   → що будувати і в якому порядку
DESIGN_SYSTEM.md    → як виглядає
EXECUTION_PLAN.md   → цей файл — коли і в якій послідовності
```

**Як користуватись цим документом:** перед початком будь-якої нової
сесії розробки — відкрити розділ "Наступний крок" внизу, він завжди
відображає поточний стан. Після завершення кроку — позначити
виконаним і посунути "Наступний крок" далі. Це живий трекер, не
статичний план, написаний один раз.

## Виправлення: тарифний гейт CRO не пускав admin/trial

Виявлено під час тестування Артемом через `/dashboard/cro` — гейт
"CRO доступний на тарифі Growth і вище" спрацював навіть під
власним акаунтом. З'ясувалось: `canUseCro()` в `croHandler.ts`
перевіряв лише `planCode === 'growth' || planCode === 'agency'`,
хоча в проєкті вже є повноцінний план `'admin'` (0016_admin_plan.sql,
`upgrade_to_admin(email)` — призначається вручну в SQL Editor,
дає `platform_role=admin` + `org_type=agency` + підписку на план
`admin` з усіма фічами `true`). Усі інші 7 місць з тим самим типом
гейту (`index.ts`, `chatHandler.ts`, `teamHandler.ts`,
`gscHandler.ts`, `seoChecker.ts`, `competitorChecker.ts`,
`fixRequestHandler.ts`) вже перевіряють
`["growth", "agency", "admin", "trial"].includes(planCode)` —
CRO єдиний був неконсистентний (найновіший модуль хвилі, гейт
писався окремо і про існуючий список не знали).

**Виправлено:** `canUseCro()` приведено до того самого списку
`planCode`, що і в інших 7 місцях. Мінімальний diff, нових
помилок tsc немає (ті самі 9 відомих, усі поза `croHandler.ts`),
`wrangler deploy --dry-run` успішно.

**Важливо для Артема — це НЕ код-фікс сам по собі:** якщо гейт
CRO досі показується під адмінським акаунтом після цього деплою,
причина не в коді, а в тому, що `upgrade_to_admin('email')` ще
не викликана для цього акаунта в Supabase SQL Editor. Це
одноразова ручна дія:
```sql
select upgrade_to_admin('твій_email@приклад.com');
```
Після виклику — `platform_role=admin` в `profiles`, організація
переведена на `agency` з безлімітними сайтами, підписка на план
`admin`. Перевірити поточний стан можна прямим запитом:
```sql
select p.platform_role, pl.code as plan_code
from profiles p
join organization_members om on om.user_id = p.id
join subscriptions s on s.organization_id = om.organization_id
join plans pl on pl.id = s.plan_id
where p.id = '<user_id>';
```

## Tasks — шостий UI-крок хвилі 3

**ВАЖЛИВО — паралельна робота двох сесій одночасно:** Артем попросив
"Tasks делай" у двох різних сесіях паралельно (можливо, ненавмисно —
одна й та сама інструкція потрапила у дві сесії). Обидві реалізували
практично ідентичний UI/API (той самий патерн проєкту), але з різним
архітектурним рішенням:
- Одна сесія: чисте MVP БЕЗ зв'язку з Agents (`agent_id`/
  `agent_run_id` nullable й непідключені) — `tasksHandler.ts`
- Друга сесія (ця): Tasks ЗІ зв'язком з Agents — content-агент
  (`agentHandler.ts`) автоматично створює задачу при старті
  (`in_progress`) і закриває її (`done`/`failed`) по завершенню
  запуску — `taskHandler.ts`

Артем explicitly підтвердив при вирішенні конфлікту: **беремо версію
зі зв'язком з Agents**, версію без зв'язку відкинуто (`tasksHandler.ts`
видалено після merge, `taskHandler.ts` — фінальний файл).

Схема (`ai_tasks`) і RLS уже існували з `0049_qorax_ai_hub.sql` — ця
сесія не чіпала жодної міграції, лише worker-логіка й UI.

**Зроблено (фінальна версія):**
- `worker/src/lib/taskHandler.ts` — `GET /api/tasks?status=`
  (список, фільтр), `POST /api/tasks` (ручне створення, `agent_id`
  завжди null для ручних задач), `PATCH /api/tasks/:id` (зміна
  статусу), `DELETE /api/tasks/:id` (owner/admin), плюс
  `createAgentTask()`/`finishAgentTask()` — окремі функції для
  виклику з `agentHandler.ts`
- **Ключова відмінність від відкинутої версії:** `agentHandler.ts`
  (`handleRunContentAgentRequest`) тепер викликає `createAgentTask()`
  одразу після створення `agent_run` (задача `in_progress`,
  `agent_id: "content"`), і `finishAgentTask()` разом із закриттям
  `agent_run` (статус `done`/`failed`, `agent_run_id` заповнюється).
  Early-return "проблемних сторінок не знайдено" відбувається ДО
  створення задачі — жодних "зависаючих" `in_progress` записів
- UI: `app/dashboard/ai/TasksTab.tsx` — фільтр за статусом, форма
  ручного створення, картка задачі. Ручна зміна статусу на "Готово"
  доступна ЛИШЕ для задач без `agent_id` (агентські задачі змінює
  тільки сам агент, не користувач вручну)
- `QoraxAiHub.tsx`: `tasks.ready = true`, заглушка "Скоро" лишається
  тільки для `automations`

**Перевірено (після merge конфлікту):** `rm -rf .next`, `tsc --noEmit`
чисто (фронт+воркер окремо, ті самі 9 відомих помилок поза новими
файлами), `npm run build` успішно, `wrangler deploy --dry-run`
успішно (519.95 KiB).

**Свідомо не зроблено (лишається за Automations):** автоматичний
запуск content-агента за розкладом (`agent_subscriptions.
schedule_cron`) — зараз агент запускається лише вручну через
`AgentsTab.tsx`. Automations має додати cron-тригер, який сам
викликає той самий `handleRunContentAgentRequest`, що вже створює
задачі — не винаходити нову логіку створення задач, лише новий
тригер запуску.

**Наступний крок:** Automations — останній порожній пункт хаба,
логічно спирається на вже готові `agents`/`agentHandler.ts`
