# Qorax — Execution Plan

Цей документ — не нова архітектура, а **послідовність дій**, що
зводить воєдино вже написані документи в один порядок роботи.
Кожен пункт нижче посилається на розділ, де рішення вже прийнято —
тут не повторюється деталізація, тільки що робити і в якому порядку.

```
PRODUCT_VISION.md   → навіщо
PLATFORM.md         → як росте платформа
DATA_MODEL.md       → структура даних
PRICING.md          → монетизація
SECURITY.md         → авторизація/RLS
API_CONTRACTS.md    → конвенція ендпоінтів
MODULE_ROADMAP.md   → що будувати і в якому порядку
DESIGN_SYSTEM.md    → як виглядає
EXECUTION_PLAN.md   → цей файл — коли і в якій послідовності
```

**Як користуватись цим документом:** перед початком будь-якої нової
сесії розробки — відкрити розділ "Наступний крок" внизу, він завжди
відображає поточний стан. Після завершення кроку — позначити
виконаним і посунути "Наступний крок" далі. Це живий трекер, не
статичний план, написаний один раз.

## Фаза 0 — Закрити технічний борг, знайдений під час написання документації

**Це має бути зроблено ПЕРШИМ, до першого рядка коду другої хвилі.**
Кожен пункт нижче — конкретна прогалина, знайдена і явно
зафіксована в SECURITY.md / API_CONTRACTS.md / PRICING.md, поки
писалась документація. Якщо пропустити цю фазу, перший же новий
модуль (Translator) успадкує ті самі діри.

### 0.1. `requireOrgAccess()` helper — ✅ ЗРОБЛЕНО (`b1ae646`)
**Джерело:** SECURITY.md розділ 5, API_CONTRACTS.md "Відомі
неузгодженості".
**Проблема:** немає єдиної функції для перевірки "юзер належить
organization_id і має роль ≥ X" на рівні Worker (RLS це не покриває
для service-role запитів). Кожен handler зараз пише цю перевірку
вручну.
**Зроблено:** `worker/src/lib/orgAuth.ts` — три функції
(`requireOrgAccess` для organization_id напряму,
`requireOrgAccessForSite`, `requireOrgAccessForProject` для
транзитивних випадків) з перевіркою ролі за ієрархією
`viewer < editor < admin < owner`. Наявні файли (rankHandler,
gscHandler) НЕ мігровано — вони і далі використовують
`getOrgIdForSite` (перевірка тільки членства, без ролі) — це
свідомо залишено для окремого проходу, не зроблено цим комітом.
**Done when:** функція існує ✅. Використання в першому новому
ендпоінті (Translator, Фаза 2) — ще не зроблено, наступний крок.

### 0.2. Консолідація `json()` helper — ✅ ЗРОБЛЕНО (`b1ae646`)
**Джерело:** API_CONTRACTS.md "Відомі неузгодженості".
**Проблема:** `json()` продубльовано в 8 файлах з різними сигнатурами.
**Зроблено:** єдина функція в `worker/src/lib/httpUtils.ts` з
перевантаженням сигнатури (`headers: Record<string,string>` АБО
`origin: string | null`), переюзовує вже наявний `corsHeaders()` з
`cors.ts`. Старі 8 копій НЕ видалено — міграція наявного коду
опційна і не зроблена цим комітом.
**Done when:** `httpUtils.ts` експортує `json()` ✅. Нові
handler-файли (Translator, Фаза 2) імпортують її — ще не зроблено,
наступний крок.

**Побічна перевірка під час цього кроку:** `npm run typecheck`
показав 9 помилок TypeScript у `worker/src` — підтверджено через
`git stash`, що всі 9 існували ДО цих змін (`index.ts`,
`formChecker.ts`), не внесені Фазою 0.1/0.2. Це окремий,
раніше не задокументований техборг — не блокер для Translator, але

### 0.3. Автоматична видача `ai_credits`

## Фаза 2 — Хвиля 2, модуль за модулем

Порядок з MODULE_ROADMAP.md: **Translator → Commerce → CRM → Social
→ CRO → Academy**, Docs — паралельно в будь-який вільний момент.
Кожен модуль проходить один і той самий цикл (уже описаний у
PLATFORM.md/пам'яті як "workflow рефакторингу", тут — адаптовано під
новий модуль):

```
1. git pull, звірити поточний стан (завжди, без винятків)
2. Прочитати відповідний розділ MODULE_ROADMAP.md (Кроки 1-5)
3. Крок 1: написати міграцію — schema звірена з DATA_MODEL.md
   (чи є organization_id/site_id/project_id шлях, чи потрібен
   новий запис у розділ 3 DATA_MODEL.md)
4. Крок 1.5 (новий, не було в roadmap): написати RLS одразу в тій
   самій міграції за шаблоном SECURITY.md розділ 4 — не окремим
   проходом пізніше
5. Крок 2: worker-логіка — ендпоінти за конвенцією API_CONTRACTS.md,
   authorization через requireOrgAccess() (Фаза 0.1)
6. Крок 3: UI — Layout за DESIGN_SYSTEM.md ("Layout модулів"),
   не винаходити новий патерн сторінки
7. Крок 4: ліміти тарифу — звірити з PRICING.md розділ 4 (там уже
   є попередній намір ліміту для кожного модуля), не вигадувати
   нову вісь ліміту без причини
8. Крок 5: критерій готовності до live з MODULE_ROADMAP.md
9. tsc --noEmit, wrangler deploy --dry-run, npm run build
10. commit, push — Артем деплоїть вручну
11. /dashboard/admin: перевести модуль у live, коли готово
    (PLATFORM.md, CHECKOUT_DISABLED)
```

Кроки 4-5 (нумерація roadmap) лишаються, як були; крок 1.5 —
уточнення, доданого цим документом, щоб RLS не писався заднім
