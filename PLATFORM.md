# Qorax — архітектура платформи

Цей документ описує платформений шар над продуктом (модулі, навігація,
захист від передчасних платежів) — окремо від документації самого
Audit-модуля (моніторинг/SEO/швидкість), яка живе в коді як звичайні
коментарі та в `/docs` на сайті.

Контекст: Qorax росте з одного продукту (моніторинг сайтів) у платформу
з кількох модулів (product vision у пам'яті проєкту). Це документ про
те, **як влаштований сам механізм росту**, а не про бізнес-план.

---

## Ключова ідея: Audit ≠ вся платформа

`sites`, `uptime_checks`, `competitor_sites` та весь існуючий моніторинг —
це один модуль (`audit`), який вже повністю працює і приносить гроші.
Він **не переписується** заради архітектурної краси.

Все інше (Sites-конструктор, AI, Content, Rank, Analytics) — це майбутні
модулі, для яких зараз існує тільки заявка на існування: рядок у
`platform_modules` зі статусом `coming_soon`, пункт у sidebar, секція на
лендингу. Жодного working-коду для них ще нема — і бути не повинно, поки
явно не почата робота над конкретним модулем.

**Правило:** ніколи не змішувати дані модуля `audit` (таблиця `sites`)
з майбутньою сутністю `projects`. Це навмисно різні таблиці, навіть коли
здається, що їх можна об'єднати.

## Схема БД (міграція 0039, 0040)

```
projects                      — майбутня сутність для Sites/Content модулів.
                                 Порожня і невикористовувана, поки конструктор
                                 не існує. НЕ плутати з `sites` (monitoring).

platform_modules               — реєстр модулів для sidebar/лендингу.
  key            text PK        напр. 'audit', 'sites', 'ai', 'content',
                                 'rank', 'analytics'
  status         enum           'live' | 'coming_soon' | 'hidden'
  href           text           куди веде пункт меню
  icon           text           ім'я lucide-іконки (рендериться в коді)
  sort_order     int

organization_module_access     — per-org оверрайд (бета-доступ)
  organization_id, module_key, enabled boolean
```

RLS: читати `platform_modules` може будь-хто авторизований (потрібно для
sidebar). Писати — тільки `is_platform_admin()` (міграція 0040, той самий
helper, що і решта admin-функціоналу).

## Як додати новий модуль у реєстр

1. `insert into platform_modules (...)` — або простіше, через
   `/dashboard/admin` → панель "Модулі платформи" (без SQL).
2. Статус за замовчуванням — `hidden`, поки немає готової сторінки, або
   `coming_soon`, якщо хочеш анонсувати заздалегідь (як зараз зроблено
   для sites/ai/content/rank/analytics).
3. Коли сторінка модуля готова: створи `app/dashboard/<slug>/page.tsx`
   (напр. `app/dashboard/sites-builder/page.tsx`) — вона автоматично
   матиме пріоритет над catch-all заглушкою
   (`app/dashboard/[module]/page.tsx`), нічого прибирати не потрібно.
4. Зміни статус модуля на `live` через адмінку — з'явиться в sidebar
   як кликабельний пункт одразу, без деплою.

## Catch-all заглушка (`app/dashboard/[module]/page.tsx`)

Перехоплює прямі переходи на URL модулів, які ще не мають власної
сторінки (`/dashboard/sites-builder`, `/dashboard/ai`, ...), і показує
консистентний екран "модуль у розробці" замість голого 404. Джерело
істини — `platform_modules`, а не хардкод шляхів.

- Шлях відповідає модулю зі статусом `coming_soon`/`hidden` → заглушка
- Шлях відповідає модулю зі статусом `live`, але власної сторінки нема
  (напр. org override для бета-тестера видав ранній доступ, а сторінку
  ще не встигли задеплоїти) → чесний 404, не бреше про готовність
- Шлях взагалі не відповідає жодному модулю → чесний 404

## CHECKOUT_DISABLED (`app/lib/checkoutFlag.ts`)

**Тимчасовий запобіжник**, поки маркетингові сторінки (лендинг, `/pricing`,
`/features`, `/about`) описують платформу з 6 модулів впевнено — без
"скоро"/"буде" — а реально працює лише `audit`. Щоб ніхто не міг
фізично оплатити через LemonSqueezy те, чого не існує, усі три checkout
entry points закриті одним прапорцем:

- `/` (лендинг, картки планів)
- `/pricing`
- `/dashboard/upgrade`

**Коли можна вимкнути:** щойно модулі, які лендинг описує як частину
платформи, реально відповідають опису (не обов'язково всі шість —
можна й раніше, якщо переписати маркетингову подачу під те, що готово
на той момент). Зміни `CHECKOUT_DISABLED` на `false` в одному файлі —
усі три точки входу знову запрацюють.

**Важливо:** якщо додається новий checkout entry point (нова сторінка з
LemonSqueezy-посиланням) — його теж треба завести під цей прапорець.
Список усіх трьох entry points вище — тримай його синхронним.

## Навігація верхнього рівня

`PlatformSidebar` (`app/dashboard/PlatformSidebar.tsx`) — постійна ліва
навігація, зараз підключена **тільки на головній `/dashboard`**.
Свідомо не зроблена як глобальний `app/dashboard/layout.tsx` — сторінки
`settings`, `admin`, `upgrade`, `referrals`, `sites/[id]` мають власні
headers і не чіпалися, щоб не ризикувати робочим кодом заради
архітектурної цілісності за один прохід.

Якщо колись вирішиш зробити sidebar глобальним для всього `/dashboard` —
це окрема, більш ризикована задача: потрібно узгодити чотири різні
headers і перевірити, що жодна сторінка не зламається.

## Маркетингові сторінки — які вже узгоджені з платформеним наративом

Лендинг (`/`), `/features`, `/about`, `/docs` — усі оновлені під
"Qorax = платформа з 6 модулів" наратив, послідовно і чесно (модулі
поза `audit` описуються на рівні цінності, не конкретних механік,
яких ще нема). `/partners` навмисно не займали — інша ціль сторінки
(реферальна воронка), не пов'язана з цим наративом.

Якщо додаєш нову маркетингову сторінку — звірся з цим списком, щоб
не створити ще один розрив у наративі.
