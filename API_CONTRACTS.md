# Qorax — API Contracts

Єдиний контракт для всіх Worker API-ендпоінтів (`qorax-api`). Новий
ендпоінт для будь-якого модуля з `MODULE_ROADMAP.md` слідує
конвенціям цього документа замість винаходу власного стилю.

**Статус:** розділи 1-4 описують факт (звірено з `worker/src/index.ts`
і handler-файлами). Розділ 5 ("Відомі неузгодженості") прямо
називає місця, де поточний код розходиться сам із собою — це
свідомо не приховано, бо мовчазне "як має бути" без згадки того, що
реально є, зробило б цей документ марним для орієнтації в наявному
коді.

---

## 1. Загальні правила

- **Немає версіонування URL.** Усі ендпоінти — `/api/...`, без
  `/v1/`, `/v2/`. Факт коду, не рекомендація на майбутнє — якщо
  колись знадобиться версіонування (напр. публічний Qorax API з
  PRODUCT_VISION.md для сторонніх розробників), це окреме
  архітектурне рішення в момент, коли з'явиться перший зовнішній
  споживач API, а не превентивна зміна зараз.
- **Роутинг — послідовний `if (url.pathname === ... && request.method
  === ...)` у `worker/src/index.ts`**, не фреймворк-роутер (Hono,
  itty-router тощо). Це свідомий вибір поточної команди — новий
  ендпоінт додається новим `if`-блоком у тому самому файлі, викликає
  handler-функцію з окремого файлу в `worker/src/lib/`. Не міняти
  цей підхід на роутер-бібліотеку без окремого рішення — це б
  вимагало переписати всі наявні ендпоінти одночасно заради
  косметики.
- **Один handler = один файл** у `worker/src/lib/<name>Handler.ts`
  (напр. `rankHandler.ts`, `teamHandler.ts`, `fixRequestHandler.ts`)
  — новий модуль повторює цю структуру, не змішує кілька непов'язаних
  ендпоінтів в одному файлі.
- **Формат відповіді:** завжди JSON через локальну функцію
  `json(data, status, originOrHeaders)`. Немає єдиної обвʼязки
  (envelope) на кшталт `{ data: ..., meta: ... }` — відповідь є
  безпосередньо об'єктом даних (`{ queries: [...] }`,
  `{ ok: true }` тощо). Новий ендпоінт продовжує цей стиль — не
  додавати envelope-обгортку, якої немає в жодному наявному
  ендпоінті.
- **Пагінація — не використовується ніде в наявному коді.** Списки
  (сайти, запити Rank, члени команди) повертаються цілком. Якщо
  новий модуль (напр. CRM-контакти, Docs-статті) матиме обсяг, що
  вимагає пагінації, — це перший прецедент, і формат
  (`?page=`/`?cursor=`) варто вирішити один раз тут, а не
  придумувати заново для кожного модуля другої хвилі. **Поточна
  рекомендація на майбутнє:** offset-пагінація (`?limit=&offset=`)
  у відповіді як `{ items: [...], total: number }` — простіше
  сумісно з наявним стилем "дані напряму в корені відповіді", ніж
  cursor-based.

## 2. Авторизація

- **Bearer JWT** — `Authorization: Bearer <token>`, виданий Supabase
  Auth. Handler дістає `userId` через звернення до
  `${SUPABASE_URL}/auth/v1/user` (патерн уже в `requireAdmin()` і
  повторюваний вручну в кожному handler'і — див. розділ 5 про
  дублювання).
- **Немає системи API Keys для зовнішніх розробників** — це
  майбутня ідея з PRODUCT_VISION.md ("AI API", "Qorax AI через
  API"), не наявний функціонал. Коли з'явиться, це нова таблиця
  `api_keys` (organization_id, hashed_key, scopes, created_at,
  last_used_at) і окремий шар автентифікації паралельно з JWT, не
  заміна його.
- **Роль перевіряється всередині handler'а**, не middleware —
  кожен handler, якому потрібен певний `member_role`, читає його
  через запит до `organization_members` (чи, для платформених
  адмін-ендпоінтів, через `requireAdmin()` з `adminAuth.ts`). Немає
  єдиного middleware, що перевіряє роль до виклику handler'а —
  це узгоджується з відсутністю роутер-фреймворку (розділ 1): немає
  місця для "middleware" в лінійному `if`-роутингу `index.ts`.

## 3. Обробка помилок

**Факт коду:** прості плоскі помилки, без коду помилки, без деталей
структури:
```json
{ "error": "Unauthorized" }
{ "error": "Not found" }
{ "error": "Некоректний запит" }
{ "error": "Максимум 50 запитів на сайт" }
```
Статус-коди вживаються послідовно:
| Код | Коли |
|---|---|
| `400` | Невалідний вхід (некоректний JSON, порушення бізнес-ліміту як-от перевищення ліміту записів) |
| `401` | Немає/невалідний токен |
| `404` | Ресурс чи organization не знайдено |
| `429` | Rate limit (де застосовується `checkRateLimit()`) |
| `500` | Помилка Supabase-запиту чи неочікуваний виняток |

Текст помилки — **змішаний українською і англійською** залежно від
handler'а (`"Unauthorized"` поруч із `"Некоректний запит"` в одному
файлі `rankHandler.ts`). Це не помилка цього документа — це поточний
стиль коду, який не варто "виправляти" масовим рефакторингом заради
консистентності; новий код може писати повідомлення українською
(ближче до аудиторії продукту), не зобов'язаний повторювати
англійські рядки старого коду.

## 4. Ендпоінти за модулями (наявні, факт)

Повний перелік не наводиться (див. `worker/src/index.ts` як
джерело правди, він змінюється частіше за цей документ) — тут
структура за групами, щоб новий ендпоінт розумів, куди в файлі
потрапляє:

| Група | Приклади | Handler-файл |
|---|---|---|
| Health/діагностика | `GET /api/health` | inline в `index.ts` |
| Audit (публічний) | `POST /api/audit`, `GET /api/report` | `auditHandler`-подібні |
| Fix requests | `POST /api/fix-request`, `GET/PATCH /api/admin/fix-requests` | `fixRequestHandler.ts` |
| Referrals | `GET /api/referrals`, `GET/PATCH /api/admin/referral-commissions` | `referralHandler.ts` |
| Team | `GET /api/team`, `POST /api/team/invite`, `PATCH/DELETE /api/team/member/:id` | `teamHandler.ts` |
| Telegram | `POST /api/telegram/webhook`, `GET /api/telegram/status` | inline/окремий файл |
| Admin (платформа) | `GET /api/admin/clients`, `POST /api/admin/run-*` (ручний запуск cron-задач) | `adminAuth.ts` + окремі |
| Rank | `GET/POST /api/sites/:id/rank/queries` (за аналогією з `rankHandler.ts`) | `rankHandler.ts` |
| AI/Content | генерація контенту, кредити | `contentGeneration.ts` |
| GSC/Analytics | `GET /api/gsc/metrics` | `gscHandler.ts` |
| Chat (Qoraxus) | чат-ендпоінт сайту | `chatHandler.ts` |

**`POST /api/admin/run-*`** — важливий існуючий патерн: ручний запуск
будь-якої cron-задачі (uptime, speed, SEO, weekly-digest, форми,
конкуренти, зламані посилання, SSL, trial-expiry) через адмін-запит,
захищений `requireAdmin()`. Нові фонові задачі для модулів другої
хвилі (напр. `run-cro-aggregate`, `run-social-publish` з
MODULE_ROADMAP.md) додаються сюди ж, за тим самим патерном — не
винаходити новий спосіб ручного запуску.

## 5. Webhooks

**Вхідні** (Qorax приймає):
- LemonSqueezy — оплата підписки, зараз вже оброблюється
  (`lemonSqueezyWebhook.ts`, variant ID matching в пам'яті)
- Telegram — `POST /api/telegram/webhook`

**Вихідні (Qorax надсилає) — НЕ реалізовано зараз як система
підписки третіх сторін.** PRODUCT_VISION.md згадує "завершення
деплою, новий лід, закінчення аудиту, оновлення позицій" як
потенційні вихідні webhook-події — це майбутня фіча для клієнтів,
які хочуть інтегрувати Qorax зі своїми системами (Zapier-подібний
сценарій), не наявний функціонал. Якщо реалізовувати: нова таблиця
`webhook_subscriptions` (organization_id, event_type, target_url,
secret) + виклик при відповідній події з тим самим ретраєм/
підписом корисного навантаження, що вже є в патерні dispatchAlert()
для Telegram/email (переюзати механізм доставки, не писати новий).

## 6. Обмеження

- **Rate limit** — `checkRateLimit()` через Cloudflare KV, зараз
  застосовується вибірково до публічних ендпоінтів (`/api/audit`,
  `/api/status/:slug`) — не глобальний middleware на кожен запит.
  Новий публічний (без автентифікації) ендпоінт ОБОВ'ЯЗКОВО
  проходить через цей helper (найкритичніший майбутній приклад —
  `POST /api/cro/track` з CRO-модуля, MODULE_ROADMAP.md, розділ 9).
- **Таймаути** — керуються лімітами самого Cloudflare Workers
  (CPU time), не налаштовуються Qorax-кодом окремо
- **Розмір запиту/файлу** — не обмежено явно в наявному коді для
  JSON body; майбутні файлові завантаження (`ai_files` з Qorax AI,
  зображення товарів Commerce) потребують явної перевірки розміру
  ПЕРЕД тим, як приймати upload — TODO зафіксувати конкретне число
  (напр. 10MB) при реалізації цих ендпоінтів, не залишати
  необмеженим

Новий ендпоінт документується тут за таким мінімальним шаблоном
(розширювати цей розділ по мірі появи нових модулів, не
переписувати весь файл):

```
POST /api/sites/:id/rank/queries
Заголовки: Authorization: Bearer <jwt>
Тіло:      { "query": "ремонт квартир київ" }
Успіх:     200 { "id": "uuid", "query": "...", "site_id": "..." }
Помилки:   401 { "error": "Unauthorized" }
           404 { "error": "Not found" }
           400 { "error": "Максимум 50 запитів на сайт" }
```

## Відомі неузгодженості (не приховувати, фіксувати як факт)

- **`json()` helper продубльований** щонайменше у 8 файлах
  (`index.ts`, `businessMetrics.ts`, `contentGeneration.ts`,
  `fixRequestHandler.ts`, `gscHandler.ts`, `rankHandler.ts`,
  `referralHandler.ts`, `teamHandler.ts`) з майже ідентичною
  реалізацією, але різною сигнатурою (`origin: string | null` в
  одних, `headers: Record<string, string>` в інших). Це кандидат
  на консолідацію в `worker/src/lib/httpUtils.ts` (файл уже існує
  для інших спільних HTTP-хелперів) — не зроблено цим документом,
  бо це зміна коду, а не документації; занесено сюди, щоб рішення
  не загубилося.
- **Мова повідомлень помилок** — змішана (розділ 3) — свідомо не
  виправляється заднім числом.
- **Немає єдиного `requireOrgAccess()` helper'а** для перевірки, що
  юзер має потрібну роль у конкретній організації на рівні Worker
  (не RLS) — уже зафіксовано як TODO в `SECURITY.md`, розділ 5;
  дублюється тут, бо це одночасно і безпекова, і контрактна
  прогалина (без неї кожен новий ендпоінт пише свою версію
  перевірки ролі, як зараз).
